## "how fast to `--help`"

I tend to work on CLIs for developers and tooling for build systems or test suites where the time from program start to end is what we're measuring. Perhaps you've noticed that running a command from a CLI may be near instant in a compiled program, but in Python, it can easily be hundreds of milliseconds: perceptible for UX, noticeable in CI/CD, and amplified by repeated calls as part of build system tooling.

Unlike in a compiled language, Python type definitions are not free (free in the sense that they were paid for during compilation _ahem, Rust_). They are code to be executed on every startup. And that includes imports of libraries and their type trees and dependents trees. We'll see in the benchmarks that (evil) metaprogramming, like decorators, metaclasses, or worse, have more of an upfront _runtime type generation_ cost than manual type definitions.

Can we get the best of everything: a Pythonic type definition style, complete static typing and [match](https://docs.python.org/3/reference/datamodel.html#object.__match_args__), with the speed of a hand-written C struct, and the startup time of a compiled extension? I _think_ so. <sup style={{fontSize: "0.7em"}}>_seriously, I'm not sure, need to do more work, but I have good preliminary data_</sup>

> Python is not _completely_ "uncompiled", and we'll be assessing both cold (no python bytecode cache) and warm (with python bytecode cache) startup times, and look at how mypyc can help with some of the costs.

But why not use a compiled language and framework like Rust + [clap](https://docs.rs/clap/latest/clap/)? I certainly do, but what can I say: I love the Python ecosystem, build tooling libraries, and the rapidly evolving type system. And I believe that the type system can continue evolvoing so that we can offload a lot of the correctness to the type checker, and reap runtime speed benefits. That's what this post is about.

> [!NOTE] Algebraic Data Types
> 
> [_sum type_](https://en.wikipedia.org/wiki/Tagged_union) is the FP term for a type union. `enum` in Rust, tagged `union` in C, `std::variant` in C++, discriminated unions in TypeScript, are common examples. A plain struct (or record, implying immutability) is a _product type_ in FP.
>
> [_pattern matching_](https://en.wikipedia.org/wiki/Pattern_matching) is the related control flow aspect. `switch` in C & C++ do not pattern match (though they can guarantee [_exhaustiveness_ at compile time](https://www.crumpledpaper.tech/2024-12-16-enum-type-safety-in-c/#exhaustiveness)), but `match` in Rust, Python, and Haskell do.
>
> If you're not feeling motivated, then [this YouTube video by No Boilerplate](https://www.youtube.com/watch?v=z-0-bbc80JM) is a great intro, though you'll have to extrapolate that this is (more or less precariously) applicable to many type systems, not just Rust - but Rust just did it well ŌØż’ĖÅ­¤”Ć (for a procedural language ­¤ś£).

Long story short, I use sum types and pattern matching everywhere, all the time, from Rust to embedded C, from Typescript to Python, from JSON to CBOR. Even if your not an FP...enthusiast, you've likely used them in Python without thinking of them as such, when reaching for `MyType | None` (an `Option` or `Maybe` type).

So, you might see something like this in every Python program I write. This example imagines that some immutable device info burned onto a ROM is versioned V1 and V2. V1 guaranteed presence of the serial number, but not the manufactured date. V2 guarantees both and adds a bootloader SHA.

```python
from typing import NamedTuple

class DeviceInfoV1(NamedTuple):
	serial_number: str
	manufactured_utc_ms: int | None

class DeviceInfoV2(NamedTuple):
	serial_number: str
	manufactured_utc_ms: int
	bootloader_sha: int

type DeviceInfo = DeviceInfoV1 | DeviceInfoV2
```

`DeviceInfo` is a _sum type_ of two _product types_,`DeviceInfoV1` and`DeviceInfoV2`, and there are only two representable states, each validated by the type system, not at runtime. Here's what the naive product type would look like:

```python
class DeviceInfo(NamedTuple):
	serial_number: str
	manufactured_utc_ms: int | None
	bootloader_sha: int | None
```

Invalid runtime states are now possible: `DeviceInfo(serial_number="abc", manufactured_utc_ms=None, bootloader_sha=123)` is a valid instance of the naive product type, but it is not a valid `DeviceInfoV1` or `DeviceInfoV2`.

> [!NOTE] The sum is smaller than the product
> 
> Where the _sum type_ version has two representable states, the _product type_ has four, two of which are invalid.
> | Variant | serial_number | manufactured_utc_ms | bootloader_sha |
> |---------|---------------|-------------------|----------------|
> | v1      | `str`           | `int`       | `None`              |
> | v2      | `str`           | `int`               | `int`            |
> | Invalid      | `str`           | `None`              | `None`           |
> | Invalid     | `str`           | `None`              | `int`             |

Using a product type instead of a sum type shifts the burden of correctness from the type system to the runtime.

> [!TIP] Result Type
>
> You can do _`Result`-like_ generic types in Python.
> 
> ```python
> from typing import NamedTuple
> 
> class Ok[V](NamedTuple):
> 	value: V
> 
> class Err[E](NamedTuple):
> 	error: E
> 
> type Result[V, E] = Ok[V] | Err[E]
> ```
> 
> A `Result` is either an `Ok` with a value of type `V`, or an `Err` with an error of type `E`. You can then pattern match on the result, and because it's covered by type checkers before runtime, it's an abstraction that gives you _compile-time correctness_ with _zero runtime cost_.
> 
> ```python
> match result:
> 	case Ok(value):
> 		print(f"Got a value: {value}")
> 	case Err(error):
> 		print(f"Got an error: {error}")
> ```
>
> Why not just `type Result[V, E] = V | E`? Because the wrapper is what we are discriminating on, not the content.
>
> _Look at [returns](https://github.com/dry-python/returns) before rolling your own FP type system in Python! Or whatever, have fun!_

## So what's the fastest struct?

Total startup ($t_{\text{startup}}$) is calculated as the fixed dependency import (the [module cost](#module-cost), $m_{\text{module}}$) plus the number of types ($N$) ├Ś the per-type definition cost ($c_{\text{type}}$).


$$
t_{\text{startup}} = m_{\text{module}} + N \cdot c_{\text{type}}
$$

The construct with the cheap dependency (`NamedTuple` leans on `typing`,
already fast and often already imported) wins at low N; the one with the cheap
per-type slope (`msgspec`) wins once N is large enough. Collapsing cold and warm
into a band per construct makes the crossover a *range*: `msgspec` overtakes
somewhere between **229 types (warm)** and **1,622 types (cold)**. Below that,
`typing.NamedTuple` is the fastest *stdlib* immutable option to start ŌĆö though the
hand-written and C-backed records below start faster still.

### NamedTuple in mypyc

I was really hoping that mypyc was going to compile NamedTuple to a native struct. Compiling the module changes *almost nothing* about the `NamedTuple`, while it transforms `native slots`:

| metric                | NamedTuple<br/>interpreted | NamedTuple<br/>mypyc | native slots<br/>interpreted | native slots<br/>mypyc |
| --------------------- | :-----------------------: | :-----------------: | :------------------------: | :------------------: |
| `isinstance(_, tuple)`| yes                       | yes                 | no                         | no                   |
| bytes / instance      | 88                        | 88                  | 64                         | 72                   |
| `__new__`             | 7 bytecodes               | 7 bytecodes         | C                          | C                    |
| `__init__`            | C                         | C                   | 9 bytecodes                | C                    |
| instance (ns)        | 138                       | 142                 | 87.5                       | 75.7                 |

The NamedTuple columns are identical: same footprint, same construct time. Its' `__new__` is still seven interpreted bytecodes inside the compiled extension module, building a tuple and handing it to `tuple.__new__`:

```text
1 RESUME                            0
  LOAD_GLOBAL                       1 (_tuple_new + NULL)
  LOAD_FAST_BORROW_LOAD_FAST_BORROW 1 (_cls, a)
  LOAD_FAST_BORROW_LOAD_FAST_BORROW 35 (b, c)
  BUILD_TUPLE                       3
  CALL                              2
  RETURN_VALUE
```

Contrast the native record. It has no `__new__` at all; its `__init__` writes
the three fields straight into their slots with `STORE_ATTR` ŌĆö no tuple, no
length field, no boxed item array. (The `Final` annotations add *zero* bytecode;
they're a pure type-checker hint, so this is byte-for-byte a plain slotted
class.)

```text
11 RESUME                   0
12 LOAD_FAST_BORROW_LOAD_FAST_BORROW 16 (a, self)
   STORE_ATTR               0 (a)
13 LOAD_FAST_BORROW_LOAD_FAST_BORROW 32 (b, self)
   STORE_ATTR               1 (b)
14 LOAD_FAST_BORROW_LOAD_FAST_BORROW 48 (c, self)
   STORE_ATTR               2 (c)
   LOAD_CONST               0 (None)
   RETURN_VALUE
```

mypyc *does* lower this `__init__` to C ŌĆö recall its 9 bytecodes became C-level in
the compiled column. But for *this* record you barely see it in the construction
numbers (87 ŌåÆ 76 ns, within run-to-run noise): the `__init__` is only three `STORE_ATTR`s, and the interpreted `timeit` loop crosses the
interpreterŌåönative boundary on every call, which caps any gain. Where compiling a
hand-written `__init__` *does* pay off is when it does real interpreted work ŌĆö
[manual record](#manual-record) routes every
field through `object.__setattr__` and drops from 222 to 78 ns once compiled, a
speedup a frozen `dataclass` can't get. NamedTuple's `__new__`, by contrast, stays
interpreted even when compiled and there's nothing for mypyc to lower at all without
breaking the tuple contract.

So, I've been right to reach for `NamedTuple` as a cheaper immutable type than `dataclass(frozen=True)`, but I was wrong to think that it was _perfectly_ efficient and compact like a C struct.

### `manual record`

A fair objection to the `Final`+slots control: it's cheap precisely because it
does *less*. It has no `__eq__`, `__hash__`, or `__repr__`, and ŌĆö as we saw ŌĆö it
isn't even immutable. Every other record here gives you all of that. So here is
the honest comparison ŌĆö a complete, genuinely-immutable hand-written record
([Brett Cannon's pattern](https://github.com/brettcannon/record-type)) with
`__slots__`, `__match_args__`, a real `__setattr__` guard, and
`__eq__`/`__hash__`/`__repr__`:

```python
class ManualRecord:
	__slots__ = ("a", "b", "c")
	__match_args__ = ("a", "b", "c")

	def __init__(self, a: int, b: int, c: int) -> None:
		object.__setattr__(self, "a", a)
		object.__setattr__(self, "b", b)
		object.__setattr__(self, "c", c)

	def __setattr__(self, _attr, _val):
		raise TypeError("immutable")

	def __eq__(self, other):
		if not isinstance(other, type(self)):
			return NotImplemented
		return self.a == other.a and self.b == other.b and self.c == other.c

	def __hash__(self):
		return hash((self.a, self.b, self.c))
```

It's been added to every chart above (as "manual record"). Three results stand
out:

- **It's cheap to import** ŌĆö ~12 ┬Ąs/type warm, no dependency, ~32├Ś cheaper than a
  frozen dataclass, because nothing is code-generated; the methods compile once
  into the `.pyc` like any hand-written class. (Cold, it has the *largest*
  compile penalty of all ŌĆö 215 ┬Ąs/type ŌĆö there's simply more source per class.)
- **It's genuinely immutable, and stays that way compiled.** Unlike `Final`, the
  `__setattr__` guard is a real runtime check, and mypyc preserves it (`o.a = 99`
  raises on the `.so`).
- **mypyc actually speeds up its construction** ŌĆö 222 ŌåÆ 78 ns, ~3├Ś ŌĆö because its
  `object.__setattr__` init is hand-written *in* the compiled module, so mypyc
  lowers it to native slot stores. The frozen dataclass can't get this (its
  `__init__` is generated at runtime and stays interpreted). The catch: the
  compiled class is *larger* (96 vs 64 bytes) ŌĆö the custom `__setattr__` costs
  mypyc its compact slotted layout.

The PyPI [`record-type`](https://pypi.org/project/record-type/) package is this
same idea as a decorator ŌĆö `@record` over a function whose signature defines the
fields. It's a lighter code generator than `@dataclass`: it only `exec`s an
`__init__`, inheriting `__eq__`/`__hash__`/`__repr__` from a shared `Record`
base, so it imports at ~96 ┬Ąs/type (a quarter of a dataclass, but still ~8├Ś the
hand-written record) and constructs at the same ~225 ns (it routes through
`object.__setattr__` too). It can't be mypyc-compiled at all ŌĆö the decorator
turns a function into a class, which mypyc rejects.

The upshot for "should this be in C?": the hand-written record is the closest
pure-Python analogue to a core `struct` type ŌĆö complete, immutable, near-zero
import cost ŌĆö and mypyc already gets its *construction* most of the way there.
What it can't deliver is msgspec's compact C layout. That gap ŌĆö a native,
immutable record with no import tax ŌĆö is exactly what C fills. `msgspec` is one
answer, but it drags a serialization library's import along with it. The next
section measures another.

### record-type (C)

The manual record marks the pure-Python ceiling: complete and immutable, with
near-zero import, but either slow to construct (222 ns) or ŌĆö once mypyc lowers its
`object.__setattr__` init ŌĆö fast (78 ns) yet *larger* (96 bytes). `msgspec.Struct`
shows C clears that ceiling: compact (64 bytes), immutable, ~62 ns construction,
~10 ┬Ąs/type. Its one catch is the cost nobody budgets for ŌĆö `import msgspec` runs
**~19 ms**, because it's a serialization library and the struct is a side effect.

So: can you get msgspec's record qualities *without* its import tax? A research
prototype on a [branch of `record-type`](https://github.com/JPHutchins/record-type/pull/1)
answers yes. It's a ~600-line C extension ŌĆö an inheritable `Record` base you
subclass exactly like `NamedTuple`:

```python
from native_record import Record

class Point(Record):       # frozen, slotted, eq/hash/repr/match_args ŌĆö all in C
	a: int
	b: int
	c: int
```

A C *metaclass* reads the class-body annotations directly (no `inspect`, no
`exec`) and builds a frozen, slotted type whose constructor is a C-level
vectorcall ŌĆö msgspec's type-creation trick, with none of its codec machinery.
Measured in the same harness as everything above:

- **Import ~0.2 ms** ŌĆö just `dlopen`-ing a small `.so`: ~95├Ś lighter than
  `msgspec`, and (uniquely here) with *no cold penalty*, since a compiled
  extension has no source to recompile ŌĆö its cold import equals its warm one.
- **Per-type ~8.6 ┬Ąs** ŌĆö on par with `msgspec`, ~9├Ś cheaper than `NamedTuple`,
  ~43├Ś cheaper than a frozen dataclass.
- **Construct ~61 ns** ŌĆö fastest tier, tied with `msgspec`: field writes happen
  in C, so there's no interpreted `__init__` and no `object.__setattr__` toll.
- **64 bytes, genuinely immutable** ŌĆö compact like `msgspec`, and `o.a = 99`
  raises. It also *inherits*: a subclass can extend the fields, which the
  `@record` decorator can't do.

Because it's cheap on import *and* per-type, it has no crossover ŌĆö it sits below
`msgspec` on the startup chart for every N Ōēź 1, where `msgspec` only overtakes
`NamedTuple` past ~229 types. It's the first *complete, immutable* option here to
win all four axes at once ŌĆö import, type-creation, construction, and memory.

Two honest caveats. It's a **research prototype**, not a release ŌĆö it lives on a
PR branch, not PyPI. And there's one real semantic limit: a class body can't
express Python's full parameter grammar (positional-only, keyword-only, `*args`,
`**kwargs`) the way `@record`'s function signature can ŌĆö fine for the
record-shaped common case, but not literally 1:1 with the decorator. (Per-type
here is measured exactly like every other construct ŌĆö module self-time ├Ę K, which
includes the ~7 ┬Ąs the bare `class` statement costs regardless ŌĆö so it is directly
comparable to the figures above.)

###  Looking upstream

Is any of this getting easier? There's active discussion of a first-class
record/struct type for Python ŌĆö Brett Cannon's
[record-type proposal](https://discuss.python.org/t/introducing-record-types-in-python/34397)
and a terser `struct Point(x: int, y: int)` spelling, with a proof-of-concept
`record` decorator now on PyPI. As proposed, though, it's mostly a *boilerplate*
win ŌĆö a concise frozen, final dataclass that generates `__slots__`, `__eq__`,
`__hash__`, and the rest ŌĆö not a new performance primitive. To be clear about
what that means: such a decorator *could* match the hand-written native class ŌĆö
it emits the same slotted layout, and the instances carry no per-decorator
overhead (the decorator runs once, at class definition, not per instance). What
it can't do is push *past* that floor. It can't make instances smaller than a
slotted object (unboxing needs types known ahead of time), and in *pure Python* it
can't deliver enforced immutability *and* fast construction together: a real
frozen class routes every field through `object.__setattr__` ŌĆö which is why the
frozen dataclass (and record-type) construct ~2.6├Ś slower than the mutable native
class, and the native class is fast at all only because `Final` isn't enforced at
runtime (so it isn't actually immutable). What *does* break that trade-off is
moving the write path to native code ŌĆö which is exactly what mypyc does to the
[hand-written manual record](#a-complete-hand-written-record) (immutable *and*
78 ns), what `msgspec` does in C, and what the [C-backed `Record`](#a-record-in-c)
does as an inheritable base in ~600 lines. A decorator can't, because its
generated `__init__` stays interpreted; that's the real reason the interesting
work lives in a compiler or in C, not in a decorator. (mypyc's remaining gap vs
hand-written C: the manual record pays for that immutability in memory ŌĆö 96 bytes
vs the 64 of `msgspec` and the C `Record`.)

The performance angle you'd actually want ŌĆö values that don't heap-allocate ŌĆö
lives on the mypyc side, in the long-standing
[record (struct) types issue](https://github.com/mypyc/mypyc/issues/841):
user-defined *unboxed* value types (Ōēł16 bytes vs 40 for a heap object), passed
around in native code and boxed only when they enter a Python container. mypyc
already does exactly this for native integers (`i64`/`i32` are unboxed value
types) ŌĆö just not yet for user-defined records. The issue has been open since
2021 with no implementation, so there's nothing to benchmark yet; treat it as a
direction, not a date. The closest thing shipping today is `msgspec.Struct` ŌĆö
that vision realized in hand-written C ŌĆö or, if you want runtime immutability
without C, the hand-written manual record compiled with mypyc: immutable and fast
to construct, just heap-allocated and a touch larger (96 bytes). The
[C-backed `Record`](#a-record-in-c) prototype shows what a *lean* shipping answer
could look like for that heap-allocated case ŌĆö `msgspec`'s struct qualities
without its import ŌĆö though genuinely unboxed values stay the mypyc issue's
domain, and it isn't on PyPI yet regardless.

All measurements were taken on a single machine: CPython 3.14.0 (installed and
managed with [uv](https://docs.astral.sh/uv/)), mypy/mypyc 2.1.0, attrs 26.1.0,
msgspec 0.21.1, and record-type 2023.1.post1, on x86_64 Linux (WSL2) with gcc
13.3. The C-backed `record-type (C)` is built from the
[branch linked above](https://github.com/JPHutchins/record-type/pull/1) (a
research prototype, not a release). Absolute numbers will differ on your hardware
and Python build ŌĆö the *relative* shape is the takeaway. Every struct carries the
same three `int` fields so the comparison is apples-to-apples.

### Interpreted vs compiled

The standard-library constructs (plain classes, slotted, `Final`-slotted,
`NamedTuple`, and the dataclass variants) live in one module that is the unit of
compilation: `mypyc containers.py` produces a `containers.*.so`. An interpreted
driver imports that module and detects which form it got by testing whether
`__file__` ends in `.so`. This mirrors how mypyc is actually used ŌĆö you compile
the *definitions* and call into them from ordinary interpreted code. The `attrs`,
`msgspec`, and both `record-type` classes are defined in the driver itself, not in
the compiled module, so there is no mypyc form to measure ŌĆö the charts and tables
leave their mypyc column empty rather than copy in the interpreted value.
(`record-type (C)` is already a compiled C extension, so mypyc has nothing to
add ŌĆö it *is* the native form.)

Worth knowing: even inside the compiled `.so`, the `@dataclass` decorator and the
`NamedTuple` metaclass still run as interpreted CPython, and the `__init__` /
`__new__` they generate stay interpreted bytecode. mypyc compiles the surrounding
module code, not the code those tools synthesize at runtime.

### Memory footprint

`sys.getsizeof` reports one object's size but doesn't follow the `__dict__`
pointer, so it understates classes that carry one. The headline figures instead
come from a bulk `tracemalloc` measurement ŌĆö allocate 200,000 instances and
subtract a same-length `[None] * n` list measured the same way, so the list's own
backing storage cancels and what remains is the instances' allocation (GC header
included):

```python
import gc, tracemalloc

def mem_per_instance(ctor, args, n=200_000):
	gc.collect()
	tracemalloc.start()
	base = [None] * n
	base_cur, _ = tracemalloc.get_traced_memory()
	objs = [ctor(*args) for _ in range(n)]
	cur, _ = tracemalloc.get_traced_memory()
	tracemalloc.stop()
	return (cur - base_cur) / n
```

Treat the per-instance figure as ┬▒one allocator alignment word; it's reported as
an integer because bytes are integers.

### Bytecode

Allocation bytecode is counted with `dis.get_instructions` on `__new__` and
`__init__` (unwrapping the `staticmethod` that wraps a NamedTuple's `__new__`),
and disassembled with `dis.dis` for the listings above. Deallocation has no
Python bytecode to count: teardown is C-level `tp_dealloc` / `tp_free` unless a
class defines a Python `__del__`, which none of these do.

### Per-instance timing

Construction and attribute access are timed with `timeit` ŌĆö the minimum of seven
repeats of 1,000,000 iterations for construction, 5,000,000 for access, reported
as nanoseconds per operation. One caveat shapes the compiled column: the `timeit`
loop is interpreted, so every iteration crosses the interpreterŌåönative boundary.
mypyc's attribute-access and call speedups land on the *compiledŌåÆcompiled* path,
so an interpreted loop reaching into a compiled class won't see them (and can even
look slightly slower) ŌĆö which is exactly why the compiled instantiation numbers
sit on top of the interpreted ones instead of below.

### Import / type-construction time

This axis needed the most care, because the obvious approach ŌĆö `timeit` on
`make_dataclass()` or `namedtuple()` ŌĆö measures the wrong thing. The dynamic
factory forms differ from the `@dataclass` and `class C(NamedTuple)` forms you
actually write (the functional `NamedTuple(...)` call understates the
class-statement form by roughly 3├Ś), and `timeit` is blind to both mypyc and the
one-time cost of importing the supporting library, since those happen before the
loop starts.

So every import number comes from a **fresh interpreter** under
`python -X importtime`, parsing the *self* time attributed to the module:

- **Per-type cost.** Generate a module of K = 200 identical-shape classes in the
  real class-statement form, import it, and read its self-time; the per-type
  figure is that self-time ├Ę 200, the median of five fresh interpreters (this is
  what the committed `importtime_sweep.py` reports). Dividing by K folds a small
  fixed per-module overhead into the figure ŌĆö a regression over K Ōłł {1, 50, 200}
  would isolate the pure slope, but for the cheapest constructs that correction is
  sub-microsecond, lost in the run-to-run noise.
- **Cold vs warm.** "Warm" imports with the `__pycache__/*.pyc` already written;
  "cold" deletes `__pycache__` first, so the source is recompiled to bytecode
  in-process. Their difference is the sourceŌåÆbytecode compile cost (tens of
  ┬Ąs/type ŌĆö ~25ŌĆō55 here, scaling with each class's source size).
- **Dependency import.** `python -X importtime -c "import LIB"` in a fresh
  interpreter gives the cumulative cost of first-importing a library. The cold
  variant points `PYTHONPYCACHEPREFIX` at an empty directory so the whole
  transitive source tree must recompile.
- **mypyc axis.** The generated module is compiled with `mypyc` and the resulting
  `.so` imported under the same harness. A compiled extension has no Python
  source to recompile, so there's no cold/warm gap ŌĆö yet its per-type creation
  cost is barely lower than interpreted, because type creation is dominated by
  CPython's `PyType_Ready`, which runs either way.

### The crossover model

The startup chart is a model, not a direct measurement: total startup is taken as
a fixed dependency import plus N ├Ś the measured per-type construction cost,
evaluated for cold and warm. The crossover is where two such lines meet ŌĆö
`N = (dep_b - dep_a) / (per_type_a - per_type_b)`. It assumes a single dependency
imported once and a linear per-type cost (both hold well here); the cold curves
roll up shared sub-dependencies, so several of these libraries imported together
cost less than the sum of their individual lines.

### Reporting

Bytes and counts are integers; timing data is quoted to three significant
figures. For the steadiest numbers, pin the CPU and run a few times ŌĆö `tracemalloc`
and small-integer interning interact, and import timings vary run to run, which is
why each is reported as the median of five fresh processes. Instantiation is the
*minimum* of seven `timeit` repeats (the conventional low-noise estimator);
treat the per-instance nanosecond figures as ┬▒10% ŌĆö the construct-to-construct
*shape* is what's robust, not the third digit.
